# 长文本溢出背景与后续计划

## 背景

TIMBot 的流式回复目前有三种承载方式：

- `text_modify`
- `custom_modify`
- `tim_stream`

这三种模式都会遇到同一个硬约束：**单条消息可承载的文本/消息体长度有限**。

这不是“是否实时”的问题，而是“是否还能继续往同一条消息里塞内容”的问题。

典型表现：

- `text_modify`：文本越来越长，最终 `modify` 失败或客户端不再正常显示
- `custom_modify`：由于 `TIMCustomElem.Data` 是 JSON 字符串，换行、引号、反斜杠都会转义，通常会比 `text_modify` 更早触发 `msg too long`
- `tim_stream`：虽然是增量发送，但 `CompatibleText` 和消息体本身也仍然有上限约束

## 已确认的事实

1. 流式数据源已经从 `onBlockReply` 切到 `onPartialReply`
2. `text_modify` / `custom_modify` 发送的是“当前完整可见文本快照”
3. 长文本问题的核心不是 chunk 拼接，而是单条消息体长度限制
4. 官方其他聊天插件对“编辑式预览流”的处理也很保守：
   - 短内容继续编辑
   - 超限后停止预览
   - 不会在预览流里做复杂的长文续写

## 当前实现

当前 TIMBot 的超限处理策略是：

1. 流式阶段持续更新同一条消息
2. 接近软上限后停止继续流式更新
3. 最终收尾时：
   - 先把当前这条流式消息改成第一段文本
   - 剩余内容按固定长度硬切
   - 以新的后续消息继续发送

这套策略的特点：

- 优点：简单、稳定、易理解
- 缺点：会在 Markdown 中间直接切开，可能破坏格式显示

## 为什么先选这个方案

之前评估过两条路线：

1. 做更讲究的 Markdown-aware splitter
2. 先用固定长度硬切，保证功能可用

当前先选第 2 条，原因是：

- 问题本质首先是“消息体装不下”
- 复杂拆分会明显增加规则和边界
- 先保证“不会因为超长直接失败”，优先级高于“每次都切得漂亮”

## 长文本优化 TODO

### P1

- 给超限续发前增加一条简短提示，说明“内容较长，后续将分段发送”
- 把当前软上限和续发分段长度做成配置项，而不是写死
- 区分 `text_modify` / `custom_modify` / `tim_stream` 的不同安全阈值

### P2

- 固定长度硬切前，先尝试向前找最近的空行再切
- 对代码块场景增加一个最小保护：避免在连续三个反引号中间切开
- 对表格场景增加最小保护：避免刚好把表头和分隔线拆开

### P3

- 评估是否需要引入更完整的 Markdown-aware splitter
- 评估是否要把超长内容切换到其他载体，而不是继续作为聊天消息发送

## 备选产品方向

如果后面不希望继续优化“消息内分段”，可改成更明确的产品策略：

- 超限后停止在 TIM 中继续发送，只发简短提示
- TIM 中仅发送摘要，完整内容改用文件/网页/文档承载
- 根据内容类型决定载体：
  - 代码：文件
  - 长文：网页或文档
  - 表格：文件或网页

这类方案的重点不是“怎么切得更漂亮”，而是“超限后换载体”。
