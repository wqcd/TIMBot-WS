# TODO: 大功能 - 支持 WebSocket 长连接

## 目标

为 TIMBot 增加基于 WebSocket 的长连接能力，用于：

- 低延迟的消息推送与状态同步
- 更自然的流式对话与多轮交互
- 后续复杂实时能力（如工具调用进度、代码执行日志流、协同编辑等）的基础设施。

## 背景

当前 TIMBot 与后台主要是基于「请求-响应」模式（如 HTTP + 轮询或短连接 RPC）：

- 每次对话问题 -> 独立请求 -> 等待完整回复
- 流式能力主要依赖 IM 自身的消息机制

这在简单问答场景下尚可，但在以下场景中存在天然限制：

- 需要持续状态同步（例如：长任务执行进度条）
- 需要低延迟的事件通知（例如：工具调用链路中的实时日志）
- 需要复杂的交互协议（例如：前端与 agent 之间的「中途打断」「重试某一步」）

WebSocket 长连接可以作为一个统一的“实时通道”，为 TIMBot 提供更强的互动性和扩展性。

## 预期形态

理想状态下：

1. 前端建立到 TIMBot 网关的 WebSocket 长连接，完成鉴权。
2. 用户在会话中发起一个问题或长任务：
   - 通过 HTTP 发起主请求（便于日志链路和重放），或
   - 直接通过 WebSocket 发送请求消息。
3. 服务端通过 WebSocket 持续推送：
   - 思考中 / 进度百分比
   - 实时 partial text / delta 文本
   - 工具调用过程中的关键事件（例如“开始运行单测”“安装依赖中”）
4. 任务结束时，通过 WebSocket 推送最终状态，并与 TIM 消息侧的“正式回复”保持一致或互补。
5. 连接稳定：
   - 前后台切换、网络波动时，客户端能自动重连并恢复必要上下文
   - 监控告警能提前发现异常（大量断线、延迟飙升等）。

## 优先级

建议优先级：中

原因：

- 属于「大功能」级别基础设施改造，投入不小
- 但一旦打好基础，后续很多体验优化（更细粒度流式、实时进度、多端协同）都会变得更容易
- 可与现有 TIM 流式能力配合使用，而不是完全替代，适合作为中长期演进方向。
